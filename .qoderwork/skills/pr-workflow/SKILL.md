# Skill: pr-workflow

## Description

在 AI Voice Drawing Tool 项目中，基于 Pull Request 驱动开发流程。每个 PR 只做一件事，保持细粒度、高质量的迭代节奏。此 Skill 规范 PR 的创建、描述、拆分和审查标准。

## When to Use

在以下场景下**必须**遵循此 Skill：

1. **新增功能**：实现任何新功能或特性
2. **修改功能**：修改现有功能的行为或实现方式
3. **修复 Bug**：修复任何缺陷或异常行为
4. **重构代码**：优化代码结构但不改变功能行为
5. **更新依赖**：添加、升级或移除第三方依赖
6. **调整文档**：对 DESIGN.md 等文档有实质性更新

## PR 核心原则

### 原则一：每个 PR 只做一件事

- 每个 PR 只实现或修改**单一功能**
- 鼓励尽可能小、粒度尽可能细的 PR
- 大功能**必须**拆分为多个独立 PR 分步提交
- 如果一个 PR 涉及超过 3 个文件的修改，考虑是否可以进一步拆分

### 原则二：PR 之间保持独立

- 每个 PR 应能独立合入，不依赖后续 PR 才能正常工作
- 如果功能确实需要多步完成，按依赖顺序排列 PR，每个 PR 是一个可运行的中间状态
- 避免在 PR 中留下"将在后续 PR 中修复"的 TODO（除非在描述中明确说明）

## PR 拆分策略

当一个需求较大时，按以下维度拆分：

| 拆分维度 | 示例 |
|---------|------|
| 按功能点 | "添加语音识别" → "添加关键词提取" → "添加 LLM 调用" |
| 按层次 | "定义数据类型" → "实现服务层" → "实现 UI 组件" |
| 按阶段 | "Phase 0 基础搭建" → "Phase 1 LLM 集成" |
| 按修复 | "修复布局溢出" → "修复坐标映射偏移" |

## PR 标题规范

标题用**一句话**描述这个 PR 做了什么，遵循以下格式：

```
<类型>: <简要描述>
```

### 类型标签

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat: 实现 LLM Function Calling 集成` |
| `fix` | 修复 Bug | `fix: 修复画布坐标映射偏移问题` |
| `refactor` | 重构 | `refactor: 抽取 LLM 服务为独立模块` |
| `style` | UI/样式调整 | `style: 重新设计侧边栏布局` |
| `docs` | 文档更新 | `docs: 更新 DESIGN.md Phase 1 完成状态` |
| `chore` | 工程配置 | `chore: 配置 Vite 多模型代理转发` |
| `perf` | 性能优化 | `perf: 优化 Canvas 重绘逻辑` |

## PR 描述规范

PR 描述**必须**包含以下三个部分，禁止留空：

### 1. 功能描述（What）

说明这个 PR 实现了什么功能或做了什么修改。

```markdown
## 功能描述
实现 LLM Function Calling 集成，替换 Phase 0 的关键词匹配方案。
通过 OpenAI SDK 调用 DeepSeek/千问/智谱的结构化工具输出，
支持 create_shape、modify_shape、delete_shape 等 5 种指令。
```

### 2. 实现方式（How）

简要说明技术方案和关键实现细节。

```markdown
## 实现方式
- 新增 `src/services/llm.ts`，封装多模型故障转移逻辑
- 修改 `App.tsx` 中的 `processVoiceCommand`，接入 LLM 调用
- 画布状态通过 `getCanvasState()` 注入 system prompt
- 坐标使用 800×600 逻辑坐标系，前端做 snap 校正
```

### 3. 测试方式（How to Test）

说明如何验证这个 PR 的功能是否正常。

```markdown
## 测试方式
1. 启动开发服务器 `npm run dev`
2. 点击麦克风按钮，说"画一个红色的圆"
3. 验证画布上出现红色圆形
4. 说"把圆移到右边"，验证圆形位置变化
5. 断开 DeepSeek API，验证自动降级到千问
```

## 禁止事项

以下情况在 PR 中是**严格禁止**的：

1. **空白描述**：PR 描述不能为空，必须包含上述三部分
2. **描述与代码不符**：描述中说的功能与实际代码修改不一致
3. **未声明的第三方依赖**：引入了新的第三方库但未在描述中说明（包括库名、版本、用途）
4. **未标注的代码复用**：从其他项目或开源代码中复用了代码但未标注来源
5. **混合不相关改动**：在一个 PR 中混入与标题无关的代码修改

## 依赖声明格式

如果 PR 引入了新的依赖，必须在描述中声明：

```markdown
## 新增依赖
| 包名 | 版本 | 用途 |
|------|------|------|
| openai | ^4.0.0 | 调用 LLM API（OpenAI 兼容格式） |
| fabric | ^6.0.0 | Canvas 2D 绘制引擎 |
```

## 代码复用标注格式

如果复用了外部代码，必须在描述中标注：

```markdown
## 代码复用
| 来源 | 文件/位置 | 复用内容 |
|------|----------|---------|
| [项目名/链接](URL) | src/utils/xxx.ts | 参考了 xxx 算法实现 |
```

## 两阶段工作流：自动 Commit + 推送提 PR

### 阶段一：自动 Commit（本地，不推送）

**触发时机**：以下任一条件满足时，**立即自动执行 commit，但不 push**：

1. **Bug 修复完成**：修复了一个 bug，经确认可正常工作，暂无遗留问题
2. **完整功能实现**：完成一个完整功能并确认实现正确、相关 bug 已修复

**Commit 规范**：

- 使用 conventional commit 格式：`<类型>: <简要描述>`
- 类型标签同 PR 标题规范（feat / fix / refactor / style / docs / chore / perf）
- 一次 commit 只包含一组相关的改动
- Commit message 用中文或英文均可，保持与对话语言一致

**执行步骤**：

```bash
git add <相关文件>
git commit -m "<类型>: <简要描述>"
```

**注意**：commit 完成后**不要自动 push**，等用户指示再推送。

### 阶段二：推送并提 PR（远程）

**触发时机**：用户要求推送（push）或要求提 PR 时触发。

**执行步骤**：

1. **检查分支**：确保在独立的功能分支上（如 `feat/llm-integration`、`fix/url-construction`）；如果还在 main 分支，先创建功能分支并推送
2. **自查代码**：确认所有 commit 都与本次 PR 主题相关，无无关改动
3. **推送分支**：`git push -u origin <分支名>`
4. **创建 PR**：使用 `gh pr create` 创建 Pull Request，标题和描述按下方规范填写
5. **自查 PR**：创建后检查 PR 页面，确认标题和描述完整准确

**PR 标题格式**：`<类型>: <简要描述>`

**PR 描述**：汇总该分支上所有 commit 的改动，按下方模板编写。

### 阶段关系示例

```
[完成功能A] → git commit -m "feat: 实现语音识别"     ← 不 push
[修复功能A的bug] → git commit -m "fix: 修复识别超时"  ← 不 push
[用户说"推送"] → git push + gh pr create              ← 推送并提 PR
[开始功能B] → git checkout -b feat/B                  ← 新分支
[完成功能B] → git commit -m "feat: 实现画布操作"      ← 不 push
[用户说"推送"] → git push + gh pr create              ← 推送并提 PR
```

## PR 模板

以下为推荐的完整 PR 描述模板：

```markdown
## 功能描述
[一句话说明这个 PR 做了什么]

## 实现方式
[技术方案和关键实现细节]

## 修改文件
[列出主要修改的文件及其改动概要]

## 测试方式
1. [具体测试步骤]
2. [预期结果]

## 新增依赖（如有）
| 包名 | 版本 | 用途 |
|------|------|------|

## 代码复用（如有）
| 来源 | 文件/位置 | 复用内容 |
|------|----------|---------|
```

## Notes

- 此 Skill 适用于所有开发阶段（Phase 0 ~ Phase 4）
- 对于紧急 Hotfix，可以适当简化描述，但不可省略"功能描述"和"测试方式"
- PR 合入后建议打 tag 标记阶段里程碑（如 `v0.1-phase0`、`v0.2-phase1`）
- 保持 PR 节奏稳定，建议每完成一个功能点就提交一个 PR，避免积攒大量修改后一次性提交
