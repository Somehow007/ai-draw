> 2026-06-13 | 基于 AI 语音绘图工具项目实践

---

## 1. 怎么实现绘图？画布 vs 多模态 LLM vs 生图 AI？

### 三种路线对比

| 维度 | 程序化画布（本项目） | 多模态 LLM 直接生成 SVG | 生图 AI（DALL-E / SD） |
|------|---------------------|----------------------|---------------------|
| **原理** | LLM 输出结构化指令 → 前端用 Fabric.js 绘制 | LLM 直接输出 SVG/Canvas 代码 | 文本 → 扩散模型 → 像素图像 |
| **可控性** | 极高：每个图形是独立对象，可增删改查 | 中：SVG 可编辑但 LLM 生成质量不稳定 | 低：输出是整张图片，无法单独操作某个元素 |
| **交互性** | 强：支持"把红色圆形改成蓝色"等精细操作 | 弱：需要重新生成整个 SVG | 无：只能重新生成整张图 |
| **延迟** | 1-3 秒（LLM 结构化输出 + 画布渲染） | 3-8 秒（LLM 生成大量 SVG 代码） | 5-15 秒（模型推理 + 图片传输） |
| **成本** | 极低（LLM token 费用） | 低（更多 token 但无额外模型） | 高（专用模型 API 费用） |
| **语音控制适配** | 天然适配：语音 → 结构化指令 → 精确操作 | 勉强适配：语音 → SVG 代码 → 难以增量修改 | 不适配：无法做"把左边的树往右移一点" |

### 本项目的选择：程序化画布

本项目选择 **Fabric.js 对象化画布 + LLM Function Calling** 路线，原因：

1. **语音控制的核心需求是"增量操作"**：用户不是要一次性生成一幅画，而是通过连续的语音指令逐步构建画面。"画一个圆" → "在圆右边画个方形" → "把圆改成红色" → "删掉方形"。这种交互模式要求画布上的每个元素都是独立可控的对象。

2. **结构化输出最稳定**：LLM 输出 `{"shape_type": "circle", "color": "red", "center_x": 400}` 远比输出 SVG 路径代码或像素图像更可靠。Function Calling 机制保证了输出格式的严格一致性。

3. **生图 AI 无法做增量编辑**：DALL-E / Stable Diffusion 输出的是整张位图，无法实现"把第二个图形往右移"这种操作。除非用 inpainting，但那会引入更多不可控因素。

### 如果重做，我会考虑什么？

- **SVG 作为中间表示**：如果不需要拖拽交互，可以考虑 LLM 直接生成 SVG。SVG 是声明式的，比 Canvas 更易于版本控制和序列化。但对于语音驱动的增量操作场景，Fabric.js 的对象模型仍然是最优选择。
- **混合方案**：基础图形用画布绘制（精确可控），复杂场景（如"画一幅山水画"）可以调用生图 AI 生成背景图片，再叠加到画布底层。

## 2. 语音识别的同音字纠错

### 问题本质

Web Speech API 的中文识别是基于通用语言模型的，它不知道用户在"绘图"这个垂直领域说话。所以：

- "矩形" → 识别为 "举行"（通用语料中"举行"频率更高）
- "梯形" → 识别为 "T型"
- "圆形" → 识别为 "园型"
- "蓝色" → 识别为 "拦色"

错误率约 **8-15%**，在垂直领域几乎是致命的。

### 本项目的双层纠错架构

```
语音识别原始文本
     │
     ▼
┌─────────────────────────────┐
│  Layer 1: 客户端词典纠错     │  pinyin-pro + 领域词汇库
│  延迟: ~5ms                 │  覆盖: 高频已知错误 (90%+)
│  方法: 拼音滑动窗口匹配     │
└──────────┬──────────────────┘
           │
           ▼
┌─────────────────────────────┐
│  Layer 2: LLM 上下文纠错    │  system prompt 引导
│  延迟: +0ms（复用 LLM 调用）│  覆盖: 剩余边界情况
│  方法: 上下文语义推理       │
└──────────┬──────────────────┘
           │
           ▼
       正确的指令文本
```

**Layer 1（客户端词典纠错）**：
- 使用 `pinyin-pro` 将 ASR 文本转为拼音
- 与预定义的领域词汇库（形状名、颜色、方位词）做拼音匹配
- 精确替换：`举行 → 矩形`、`T型 → 梯形`
- 模式替换：`X型 → X形`、`X涩 → X色`（覆盖一类规律）

**Layer 2（LLM 上下文纠错）**：
- 在 system prompt 中列出常见错误映射
- LLM 根据上下文自动理解正确意图
- 不额外告知用户纠正了什么（体验更流畅）

### 为什么不选其他方案？

| 方案 | 不选原因 |
|------|---------|
| Whisper API | 小模型中文识别质量反而不如 Web Speech API，且增加 1-3 秒延迟 |
| SenseVoice/FunASR | 需要部署服务端，增加架构复杂度和成本 |
| ChineseErrorCorrector (4B 模型) | 需要 GPU 服务器，杀鸡用牛刀 |
| 简单字符串替换 | 无法覆盖所有变体，维护成本高 |

### 如果要做到更好

1. **热词/Hotword 功能**：部分 ASR 引擎（如 Azure Speech、Google Cloud Speech）支持热词配置，可以在识别阶段就提高领域词汇的权重。Web Speech API 不支持此功能。
2. **Whisper large-v3 + 微调**：如果用 Whisper 大模型并用绘图领域数据微调，识别准确率会显著提升。但成本和延迟都会增加。
3. **多轮确认**：对于低置信度的识别结果，可以让 LLM 反问确认："您是想画一个矩形吗？"

## 3. LLM 的响应怎么画？如何规定响应格式？

### 核心原则：结构化、可执行、可验证

LLM 的响应必须是**机器可直接执行的结构化指令**，而不是自然语言描述。

### 本项目的方案：OpenAI Function Calling

```json
// LLM 返回的 tool call 示例
{
  "function": {
    "name": "create_shape",
    "arguments": {
      "shape_type": "circle",
      "color": "#FF0000",
      "center_x": 400,
      "center_y": 300,
      "size": 80
    }
  }
}
```

前端解析 JSON → 调用 `canvas.createShape(args)` → Fabric.js 渲染。

### 工具设计的核心原则

**1. 参数尽量少而精**

```
❌ 不好的设计：create_shape(type, x, y, width, height, radius, rx, ry, points, ...)
✅ 好的设计：create_shape(shape_type, center_x, center_y, size, color)
```

用 `size` 统一表示尺寸（圆形=半径，矩形=边长），而不是让 LLM 分别输出 width/height/radius。减少 LLM 的认知负担，降低出错概率。

**2. 坐标系要明确**

- 告诉 LLM 逻辑坐标系是 800×600
- 前端按实际画布尺寸做比例映射
- LLM 始终在固定坐标系中思考，不需要关心窗口大小

**3. 用枚举约束值域**

```json
"shape_type": {
  "type": "string",
  "enum": ["circle", "rectangle", "triangle", "ellipse", "line", "star"]
}
```

枚举比自由文本可靠 10 倍。LLM 不会输出 `"shape_type": "五角形"` 这种前端无法处理的值。

**4. 一次调用可以返回多个 tool call**

用户说"画一个圆和一个方形"，LLM 返回两个 `create_shape` 调用。前端顺序执行即可。不需要设计"批量创建"工具。

### 如果不用 Function Calling？

| 方案 | 优劣 |
|------|------|
| **JSON Mode**（让 LLM 直接输出 JSON） | 可行，但格式稳定性不如 Function Calling。需要自己定义 schema 并做错误恢复。 |
| **自然语言解析**（LLM 输出"我要在中间画一个红色圆形"，前端用正则解析） | 极其脆弱，不同表述会导致解析失败。不推荐。 |
| **代码生成**（LLM 输出 JavaScript 代码） | 安全风险大，且代码执行错误难以恢复。不推荐。 |

### 最佳实践总结

```
1. 用 Function Calling 而非自由格式输出
2. 参数名用 snake_case，含义明确
3. 可选参数给默认值，减少必填项
4. 用 enum 约束值域
5. 工具数量控制在 5-8 个（过多会降低 LLM 选择准确率）
6. 在 description 中写清楚每个参数的含义和默认值
```

## 4. 怎么提升响应速度？

### 延迟瓶颈分析

```
用户说完话
  │
  ├── 语音识别延迟 ....... 200-500ms（Web Speech API 流式，几乎不可避免）
  │
  ├── 文本纠错 ........... ~5ms（可忽略）
  │
  ├── LLM 推理延迟 ....... 1-3s（主要瓶颈！）
  │     ├── 网络传输 ....... 100-300ms
  │     ├── 模型推理 ....... 500-2000ms
  │     └── 响应传输 ....... 100-300ms
  │
  ├── 绘图渲染 ........... <50ms（可忽略）
  │
  └── 总计 ............... 1.5-4s
```

### 已实现的优化

1. **L0 快速指令**：清空、撤销等高频操作走关键词匹配，不走 LLM，延迟 < 50ms
2. **多模型故障转移**：DeepSeek（最快）→ Qwen → Zhipu，优先用延迟最低的模型
3. **流式语音识别**：Web Speech API 的 `interimResults` 让用户在说话时就能看到实时文字，减少"等待感"

### 可进一步优化的方向

#### A. 流式输出（最高优先级）

当前方案：等待 LLM 完整返回后才开始执行。

优化方案：使用 `stream: true`，LLM 返回第一个 tool call 时就开始执行绘图，后续 tool call 继续追加。

```typescript
// 伪代码
const stream = await client.chat.completions.create({
  model: 'deepseek-chat',
  messages,
  tools,
  stream: true,  // 关键
});

for await (const chunk of stream) {
  // 解析增量 tool call，收到完整参数后立即执行
  if (isToolCallComplete(chunk)) {
    executeToolCall(parseToolCall(chunk));
  }
}
```

预期提升：**复合指令的首个图形出现时间减少 50-70%**。

#### B. 指令缓存

对高频指令（如"画一个红色圆形"）缓存 LLM 响应。下次遇到相似指令时直接复用缓存，跳过 LLM 调用。

```typescript
const cacheKey = hash(correctedText + canvasStateSignature);
const cached = instructionCache.get(cacheKey);
if (cached) {
  executeToolCalls(cached);  // 跳过 LLM
  return;
}
```

难点：画布状态不同，同一个指令的目标对象 ID 不同。需要设计智能缓存键。

#### C. 预取/预测

当用户说"画一个红色的..."时，在 ASR 还在识别后续内容时，就预加载 LLM 连接。

#### D. 模型选择

| 模型 | 推理延迟 | 中文能力 | 推荐场景 |
|------|---------|---------|---------|
| DeepSeek-V3 | ~800ms | 良 | 默认首选 |
| Qwen-Turbo | ~600ms | 良 | 速度优先场景 |
| GLM-4-Flash | ~500ms | 中 | 兜底 |
| GPT-4o-mini | ~1200ms | 优 | 复杂推理 |

选择延迟最低的模型作为默认，复杂指令时切换到能力更强的模型。

#### E. 前端感知优化

- 语音识别阶段就显示"思考中..."动画，让用户感觉系统在并行工作
- 图形创建动画（scale 0→1）让"等待"变成"期待"
- TTS 播报执行结果，让用户不需要盯着屏幕

## 5. 复杂图形怎么画？蒙娜丽莎？奶龙？无头骑士？

### 诚实的回答：当前方案画不了

本项目的程序化画布方案（Fabric.js + 基本图形）能画的是**几何图形组合**级别的绘图：

```
✅ 能画好的：
  房子 = 矩形 + 三角形 + 小矩形
  太阳 = 圆形 + 线段
  花朵 = 圆形 + 椭圆 + 线段
  简笔小车 = 矩形 + 圆形
  交通信号灯 = 矩形 + 三个圆形

❌ 画不好的：
  蒙娜丽莎（需要像素级精确控制）
  奶龙（特定 IP 形象，需要精确造型）
  无头骑士骑幽灵马（复杂姿态 + 细节）
```

### 为什么画不了？

**根本原因：LLM 的空间推理能力有限**

LLM 擅长语言推理，但不擅长精确的空间布局。让 LLM 通过 `create_shape` 调用画出蒙娜丽莎，相当于让它用几百个圆形、矩形、椭圆的坐标和颜色来"拼"出蒙娜丽莎——这对当前的 LLM 来说是不可能的。

### 如果真的要做，有哪些路线？

#### 路线 A：调用生图 AI 生成图片 → 叠加到画布

```
用户："画一个蒙娜丽莎"
    │
    ▼
LLM 判断：这是复杂图像，无法用基本图形组合
    │
    ▼
调用 DALL-E / Stable Diffusion API
    │
    ▼
获取生成的图片 URL
    │
    ▼
Fabric.js fabric.Image.fromURL() 加载到画布
    │
    ▼
用户仍可继续语音操作："把蒙娜丽莎放大" / "在蒙娜丽莎右边画个框"
```

**优点**：效果好，能画任何东西
**缺点**：
- 需要生图 AI API（成本）
- 延迟 5-15 秒
- 生成的图片是一个整体，无法做"把蒙娜丽莎的手往左移"这种精细操作
- 风格一致性难以保证（每次生成的风格可能不同）

#### 路线 B：SVG 路径生成

```
用户："画一个奶龙"
    │
    ▼
LLM 联网搜索"奶龙 SVG"或参考图片
    │
    ▼
多模态 LLM 分析图片 → 生成 SVG path 数据
    │
    ▼
Fabric.js 加载 SVG 路径到画布
```

**优点**：SVG 是矢量图，可缩放不失真，可编辑单个路径
**缺点**：
- 当前 LLM 生成复杂 SVG 路径的能力非常有限
- 需要大量 prompt engineering 和 few-shot 示例
- 质量不可控

#### 路线 C：预置素材库 + 语音检索

```
预先准备一个素材库：
  - 简笔画素材（房子、树、花、动物...）
  - IP 形象素材（需授权）
  - 图标素材

用户："画一个奶龙"
    │
    ▼
LLM 理解意图 → 搜索素材库中匹配项
    │
    ▼
找到最接近的素材 → 加载到画布
    │
    ▼
用户可继续修改："把它放大" / "换个颜色"
```

**优点**：质量可控，延迟低
**缺点**：素材库有限，无法覆盖所有请求

#### 路线 D：混合方案（推荐）

```
指令复杂度判断（LLM 或前端规则）
     │
     ├── 简单几何组合 → 程序化画布（当前方案）
     │     "画一个红圆和蓝方形" → create_shape × 2
     │
     ├── 中等复杂度 → 增强型分解（更多图形 + 路径）
     │     "画一个小狗" → 分解为 10-15 个基本图形
     │
     └── 高复杂度 → 调用生图 AI
           "画蒙娜丽莎" → DALL-E API → Image 叠加到画布
```

**核心思路**：在 system prompt 中增加一个 `assess_complexity` 工具，让 LLM 先评估指令复杂度，再决定用哪种方式执行。

### 对于"需要联网才知道的内容"

用户说"画一个无头骑士骑着幽灵马"，LLM 可能不知道"无头骑士"具体长什么样。解决方案：

1. **LLM 联网搜索**（如果模型支持）：搜索"无头骑士"的描述和图片 → 提取视觉特征 → 分解为基本图形
2. **多模态模型**：给模型看参考图 → 让它描述画面构成 → 生成绘图指令
3. **用户描述补充**：LLM 反问 → "无头骑士是指穿着盔甲、没有头颅、骑着马的骑士吗？请描述更多细节"
4. **直接调用生图 AI**：这类复杂场景直接用 DALL-E 生成效果最好

### 总结

| 复杂度 | 方案 | 示例 |
|--------|------|------|
| L1 基本图形 | Fabric.js 程序化绘制 | 圆、方、三角、线段 |
| L2 简单组合 | LLM 分解 + 多次 create_shape | 房子、太阳、花朵 |
| L3 中等复杂 | 增强分解 + SVG 路径 | 简笔动物、交通工具 |
| L4 高复杂度 | 调用生图 AI + 画布叠加 | 蒙娜丽莎、IP 角色 |
| L5 未知概念 | 联网搜索 + 生图 AI | 特定角色、抽象概念 |

**本项目的定位是 L1-L2 级别的语音绘图工具**，这是一个明确的产品边界。L3+ 的能力需要引入额外的 AI 模型和基础设施，属于完全不同的技术栈。在有限的开发时间和资源下，把 L1-L2 做到极致（准确率 > 95%、延迟 < 2 秒）比追求 L4-L5 的"能画但不能画好"更有价值。

## 附：项目技术决策总结

```
选了什么：
  ✅ Fabric.js 对象化画布（可控性 > 表现力）
  ✅ LLM Function Calling（结构化 > 自由文本）
  ✅ 双层 ASR 纠错（低成本 + 高覆盖）
  ✅ 多模型故障转移（可用性 > 单一最优）
  ✅ 800×600 逻辑坐标系（LLM 友好 + 前端适配）

没选什么：
  ❌ 生图 AI（不可控、不可编辑、高成本）
  ❌ LLM 直接生成 SVG/代码（不稳定、不安全）
  ❌ 服务端 ASR（重、贵、增加架构复杂度）
  ❌ 本地大模型纠错（需要 GPU、开发成本高）
  ❌ 追求画复杂图像（超出产品定位）
```
