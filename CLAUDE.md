# AI Voice Drawing Tool — 项目规范

## PR 工作流规范

### 核心原则

1. **每个 PR 只做一件事**：每个 PR 只实现或修改单一功能，鼓励细粒度拆分
2. **大功能必须拆分**：按功能点、层次、阶段或修复维度拆为多个独立 PR
3. **PR 之间保持独立**：每个 PR 应能独立合入，不依赖后续 PR 才能正常运行

### PR 标题格式

```
<类型>: <简要描述>
```

类型标签：`feat`（新功能）、`fix`（修复）、`refactor`（重构）、`style`（UI/样式）、`docs`（文档）、`chore`（工程配置）、`perf`（性能优化）

### PR 描述必须包含三部分

1. **功能描述（What）**：这个 PR 做了什么
2. **实现方式（How）**：技术方案和关键实现细节
3. **测试方式（How to Test）**：如何验证功能是否正常

### 禁止事项

- 空白描述
- 描述与代码修改不一致
- 未声明的第三方依赖（包名、版本、用途）
- 未标注的外部代码复用（来源、文件、复用内容）
- 混合不相关的代码改动

### 两阶段工作流：自动 Commit + 推送提 PR

#### 阶段一：自动 Commit（本地，不推送）

**触发时机**：以下任一条件满足时，立即自动执行 commit，但不 push：

1. **Bug 修复完成** — 修复了一个 bug，经确认可正常工作，暂无遗留问题
2. **完整功能实现** — 完成一个完整功能并确认实现正确、相关 bug 已修复

**Commit 规范**：
- 格式：`<类型>: <简要描述>`（类型同 PR 标题标签）
- 一次 commit 只包含一组相关改动
- Commit message 语言与对话语言一致

**注意**：commit 完成后不要自动 push，等用户指示再推送。

#### 阶段二：推送并提 PR（远程）

**触发时机**：用户要求推送（push）或要求提 PR 时。

**执行步骤**：
1. 确保在独立功能分支上（如 `feat/xxx`、`fix/xxx`）；如果在 main 则先创建分支
2. 自查所有 commit 都与本次 PR 主题相关
3. `git push -u origin <分支名>`
4. `gh pr create` 创建 PR，标题和描述按规范填写
5. 检查 PR 页面确认完整

**阶段关系**：完成即 commit（本地）→ 用户说推送 → push + 提 PR

### PR 描述模板

```markdown
## 功能描述
[一句话说明这个 PR 做了什么]

## 实现方式
[技术方案和关键实现细节]

## 修改文件
[列出主要修改的文件及改动概要]

## 测试方式
1. [具体测试步骤]
2. [预期结果]

## 新增依赖（如有）
| 包名 | 版本 | 用途 |
|------|------|------|

## 代码复用（如有）
| 来源 | 文件/位置 | 复用内容 |
|------|----------|---------|
```
