# AI 语音绘图工具 — 设计开发方案

## 1. 项目概述

### 1.1 项目目标

开发一款纯语音控制的绘图工具。用户不能使用鼠标或键盘，仅通过语音指令完成图形绘制创作。系统需要准确理解用户的自然语言指令，将其转化为画布上的图形操作，并提供即时的视觉与语音反馈。

### 1.2 设计原则

- **稳字当头**：核心链路（语音识别 → 指令解析 → 绘图执行）必须极其稳定
- **分层递进**：基础指令绝对可靠，高级能力逐步叠加
- **高水平组装**：不做底层技术攻关，选择成熟开源库与云服务进行组装
- **工程量聚焦**：把精力集中在指令理解层和交互体验层

### 1.3 核心竞争力

指令理解层（LLM Function Calling + 指令系统设计）和交互体验层（语音状态可视化 + 容错纠错机制）是本项目的技术深度所在。

## 2. 技术栈选型

| 层级 | 选型 | 版本 | 选型理由 |
|------|------|------|---------|
| 前端框架 | Vite + React + TypeScript | Vite 6 / React 18 | Vite 极速开发体验；React 组件化解耦语音层、绘图层、UI 层；TypeScript 为指令系统提供强类型保障 |
| 绘图引擎 | Fabric.js | v6 | 对象化画布模型，天然适合程序指令控制 |
| 语音识别 | Web Speech API（主）/ Whisper API（备） | 浏览器原生 | 零成本、实时流式、低延迟 |
| 指令解析 | OpenAI 兼容 API + 多模型故障转移 | — | Function Calling 输出严格结构化 |
| 语音反馈 | Web Speech Synthesis API | 浏览器原生 | 零成本、零配置 |

### 2.1 绘图引擎选型详细对比

| 维度 | Fabric.js ✅ | Konva.js | 原生 Canvas API |
|------|------------|----------|----------------|
| 对象管理 | 内建完整对象模型，支持选择、分组、z-index | 需要 Layer/Group 手动管理 | 无，需完全自行实现 |
| 序列化 | toJSON() / loadFromJSON() 原生支持 | 需要自行实现 | 不适用 |
| 编程式操作 | 通过 API 直接创建/修改/删除对象 | 类似 | 每次操作需手动重绘 |
| 撤销/重做 | 可基于 JSON 快照轻松实现 | 可行但更复杂 | 极其困难 |
| 图形类型 | 矩形、圆、椭圆、三角形、多边形、路径、文本、图片 | 类似，路径支持较弱 | 需逐个手写 |

**结论**：Fabric.js 的"对象化画布"理念意味着画布上每个图形都是独立 JS 对象，天然适合通过程序指令（语音解析后的结构化命令）来控制。

### 2.2 语音识别方案对比

| 维度 | Web Speech API（主） | Whisper API（备） |
|------|---------------------|------------------|
| 成本 | 零 | 按音频时长计费 |
| 延迟 | 200-500ms（流式） | 1-3s（需录制上传） |
| 准确率 | 安静环境 > 95% | 嘈杂环境更优 |
| 适用场景 | 演示环境良好时 | 现场噪音大时 |

## 3. LLM 集成方案（多模型故障转移）

### 3.1 模型提供商与优先级

| 优先级 | 提供商 | 模型 | API Base URL | 选型理由 |
|--------|--------|------|-------------|---------|
| P0 | DeepSeek | deepseek-chat | `https://api.deepseek.com` | API 格式完全兼容 OpenAI，零迁移成本；function calling 稳定；价格极低 |
| P1 | 通义千问 | qwen-max | `https://dashscope.aliyuncs.com/compatible-mode/v1` | 中文理解能力国内顶尖；OpenAI 兼容模式降低适配成本 |
| P2 | 智谱 GLM | glm-4-flash | `https://open.bigmodel.cn/api/paas/v4` | 有免费额度，作为兜底方案 |

### 3.2 故障转移策略

```
请求发出
  │
  ▼
DeepSeek API 调用 ──成功──→ 返回结果
  │ 失败（网络超时 / API 错误 / 速率限制）
  ▼
通义千问 API 调用 ──成功──→ 返回结果
  │ 失败
  ▼
智谱 GLM API 调用 ──成功──→ 返回结果
  │ 失败
  ▼
返回友好错误 → 前端提示用户重试
```

实现方式：统一的 `LLMService` 类封装所有提供商调用，使用 OpenAI SDK 兼容格式。每次请求按优先级依次尝试，当前提供商失败时自动降级到下一个。记录失败原因用于调试。

### 3.3 Function Calling 工具设计

系统定义以下工具供 LLM 调用：

**create_shape** — 创建图形
- shape_type: circle / rectangle / square / triangle / ellipse / line / arrow
- color: 填充颜色（hex 或颜色名）
- center_x, center_y: 中心坐标（像素值）
- size: 尺寸（圆形为半径，矩形为边长）
- stroke_color: 边框颜色（可选）
- opacity: 透明度 0-1（可选，默认 1）

**modify_shape** — 修改已有图形
- target_id: 目标对象 ID（从画布状态上下文中获取）
- color: 新颜色（可选）
- center_x, center_y: 新位置（可选）
- scale: 缩放倍数（可选）
- rotation: 旋转角度（可选）
- opacity: 透明度（可选）

**delete_shape** — 删除图形
- target_id: 目标对象 ID

**clear_canvas** — 清空画布（无参数）

**query_canvas** — 查询画布状态
- query_type: count / largest / smallest / colors / by_type

### 3.4 坐标映射策略

> 更新于 2026-06-12：画布改为响应式设计，坐标系统相应调整。

**方案：LLM 直出像素坐标 + 前端按比例映射 + 后处理修正**

- System Prompt 中告知 LLM 逻辑画布尺寸（800×600）作为参考坐标系
- LLM 输出基于此逻辑坐标系的绝对像素值
- 前端按实际画布尺寸做比例映射：`实际X = (LLM输出X / 800) * 画布实际宽度`
- 前端做一层后处理修正：
  - **网格吸附**：坐标接近画布中心/四分点/边缘时自动对齐
  - **边界修正**：确保图形不会超出画布范围
  - **最小尺寸保护**：防止 LLM 输出过小的尺寸

**优势**：LLM 始终使用固定的逻辑坐标系思考，前端负责适配不同窗口尺寸。

### 3.5 目标定位策略

每次请求 LLM 时，将当前画布状态注入 System Prompt：

```
当前画布对象:
[
  { id: "obj_1", type: "circle", color: "#FF0000", x: 400, y: 300, radius: 80 },
  { id: "obj_2", type: "rectangle", color: "#0000FF", x: 200, y: 150, width: 120, height: 80 }
]
```

LLM 通过对象 ID 引用目标。例如用户说"把红色的圆改成蓝色"，LLM 看到上下文中 obj_1 是红色圆形，在 tool call 中输出 `target_id: "obj_1"`。前端根据 ID 找到对象执行修改。

定位策略优先级：精确匹配（颜色+形状）→ 最近创建 → 模糊描述（最大/最左）→ 确认机制（匹配不唯一时询问）。

## 4. 指令系统设计

### 4.1 指令层级

**L0 — 画布控制指令（关键词匹配，不走 LLM，延迟 < 50ms）**

| 指令 | 操作 |
|------|------|
| "清空画布" | canvas.clear() |
| "撤销" / "回退" | undo() |
| "重做" | redo() |

**L1 — 基础绘图指令（LLM Function Calling，核心必保）**

| 指令类型 | 示例 | 对应工具 |
|---------|------|---------|
| 创建 | "画一个红色的大圆形" | create_shape |
| 创建+定位 | "在中间画一个蓝色矩形" | create_shape |
| 修改颜色 | "把红色的圆改成蓝色" | modify_shape |
| 修改大小 | "把矩形放大两倍" | modify_shape |
| 修改位置 | "把三角形移到右上角" | modify_shape |
| 删除 | "删掉最后一个图形" | delete_shape |

**L2 — 复合与空间指令（Phase 2 实现）**

| 指令类型 | 示例 | 处理方式 |
|---------|------|---------|
| 复合 | "画一个红圆和一个蓝方形" | LLM 返回多个 tool call |
| 空间关系 | "在圆形右边画一个方形" | LLM 计算相对坐标 |
| 查询 | "画布上有几个图形" | query_canvas |

**L3 — 智能推理指令（Phase 4 可选）**

| 指令类型 | 示例 | 处理方式 |
|---------|------|---------|
| 模糊理解 | "画一个好看的房子" | LLM 组合多个基础图形 |
| 风格变换 | "用暖色调重绘所有图形" | 批量 modify_shape |

## 5. 语音交互设计

### 5.1 语音状态可视化

| 状态 | 视觉反馈 | 说明 |
|------|---------|------|
| 待机 | 呼吸灯效果麦克风图标 | 系统正在监听 |
| 识别中 | 波形动画 + 实时字幕 | 正在识别语音 |
| 思考中 | 加载动画 + "正在理解..." | LLM 正在解析指令 |
| 执行成功 | 绿色勾号 + 图形出现动画 | 绘图完成 |
| 执行失败 | 红色提示 + TTS 引导 | 引导用户重新表述 |

### 5.2 确认机制

破坏性操作（清空画布、批量删除）需二次确认：TTS 播报"确定要清空画布吗？说'确定'来确认"。

### 5.3 指令历史面板

屏幕右侧固定 280px 侧栏显示指令历史，每条包含：用户说了什么 → 系统理解为什么 → 执行了什么操作。侧栏内独立滚动，不影响画布区域。演示时是极好的佐证，证明系统在"理解"而非"猜测"。

### 5.4 界面布局设计

> 更新于 2026-06-12：经实际测试后重新设计布局，解决指令历史挤压画布的问题。

采用"左画布 + 右侧栏"分栏布局，确保各元素互不干扰：

```
┌──────────────────────────────────────────────┐
│  AI 语音绘图工具              (48px 标题栏)   │
├──────────────────────────┬───────────────────┤
│                          │  🎤 语音控制      │
│                          │  (反馈信息)       │
│      画布区域            │───────────────────│
│      (自适应填充)        │  指令历史         │
│                          │  (独立滚动)       │
├──────────────────────────┤  ✅ "画圆" → ...  │
│ 试试说: [画圆] [矩形]   │  ❌ "xxx" → ...   │
└──────────────────────────┴───────────────────┘
```

- **画布区域**：flex: 1 自适应填充，使用 ResizeObserver 检测容器尺寸，图形尺寸按比例计算
- **右侧栏**：固定 280px 宽度，包含语音控制卡片和指令历史面板
- **指令历史**：flex: 1 占满侧栏剩余空间，内部 overflow-y: auto 独立滚动
- **提示条**：画布底部，展示可用指令示例

### 5.5 语音识别同音字纠错

> 更新于 2026-06-12：Phase 0 阶段使用同音字映射表作为临时方案，Phase 1 接入 LLM 后将自然解决此问题。

**问题**：Web Speech API 中文识别常出现同音字错误，如"矩形"→"举行"、"圆"→"园"。

**Phase 0 临时方案**：正则映射表预处理，覆盖形状、颜色、动作、位置等常见同音字。

**Phase 1 根本方案**：LLM 天然理解上下文语义，"画一个蓝色举行"会被正确理解为"画一个蓝色矩形"，无需任何映射规则。这也是尽快推进到 Phase 1 的核心动力之一。

```
         ▼

         ▼
```

### Phase 1：核心链路打通（2-3 天）

**目标**：跑通完整的"语音 → LLM 解析 → 绘图执行"链路，支持 L0 + L1 基础指令。

**具体工作**：
- 实现 Command Parser 模块，接入 LLM Function Calling（含多模型故障转移）
- 实现 Command Executor 模块，将结构化指令翻译为 Fabric.js API 调用
- 实现 ObjectResolver 模块，支持按 ID 定位目标对象
- 实现 L0 指令（清空、撤销、重做）—— 基于关键词匹配
- 实现 L1 创建/修改/删除类指令
- 实现语音状态可视化

**验收标准**：
- 能正确执行 L1 级别语音指令（成功率 > 90%）
- 语音到绘图的端到端延迟 < 3 秒
- 多模型故障转移可正常工作

### Phase 2：智能增强（2-3 天）

**目标**：支持复合指令和空间关系推理。

**具体工作**：
- 复合指令拆解：一句话多操作
- 空间关系指令：相对定位
- 查询指令：画布状态查询
- 目标确认机制
- 指令历史面板
- 绘图执行动画

**验收标准**：
- 复合指令正确拆解并依次执行
- 空间关系指令正确计算相对位置（±20px 偏差）
- 指令历史面板完整记录每条指令

### Phase 3：体验打磨（2 天）

**目标**：针对比赛演示场景专项优化。

**具体工作**：
- Whisper API 备选通道 + 自动降级
- 延迟体验优化（流式识别显示 + LLM 流式输出）
- 演示模式设计
- 错误恢复机制
- UI 美化

**验收标准**：
- 演示脚本可连续执行 15+ 条指令无中断
- 错误优雅恢复，不会卡死或白屏
- 整体演示 3-5 分钟内完成

### Phase 4（可选）：高阶能力（1-2 天）

方向 B — 智能纠错引导（"画一个图形" → "您想画什么形状呢？"）

## 8. 风险控制

| 风险 | 应对方案 |
|------|---------|
| 现场网络不稳定 | 本地缓存常见指令解析结果；准备手机热点 |
| 现场噪音导致识别率低 | 切换 Whisper API 模式；准备外置降噪麦克风 |
| 大模型 API 不可用 | 多模型故障转移机制（DeepSeek → 千问 → 智谱） |
| 复合指令解析错误 | 指令历史面板展示系统理解；用撤销回退 |
| API Key 泄露 | Vite dev proxy 转发，Key 仅在 .env 文件中 |

## 9. 演示脚本（3 分钟参考）

```
第一步 — 基础创建（L1）：
  "画一个大的蓝色圆形在画布中间"         → 蓝色大圆出现
  "在圆形的右边画一个红色正方形"         → 红色方形出现在右侧
  "在左下角画一个绿色三角形"             → 绿色三角出现

第二步 — 属性修改（L1）：
  "把蓝色的圆改成橙色"                   → 圆形变色
  "把正方形缩小一半"                     → 方形缩小

第三步 — 复合指令（L2）：
  "在上面再画三个不同颜色的小圆形排成一排" → 三个圆自动排列

第四步 — 查询（L2）：
  "画布上一共有几个图形"                  → TTS/面板显示数量

第五步 — 撤销与容错：
  "撤销上一步"                           → 操作撤销
```
